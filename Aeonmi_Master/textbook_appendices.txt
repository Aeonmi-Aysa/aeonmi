# APPENDIX A: Complete CLI Reference

### SECTION: Overview
BODY: The `aeonmi` binary is the unified entry point for the entire Aeonmi ecosystem. Every capability — from running a single .ai file to interacting with Mother AI to minting NFT metadata — flows through this one command. The CLI is built with Clap and follows a subcommand pattern: `aeonmi <subcommand> [flags] [args]`. Global flags apply to every subcommand and must be placed before the subcommand name.

### SECTION: Global Flags
TABLE_HEADER: Flag | Type | Description
TABLE_ROW: --pretty-errors | boolean | Enable colorized, human-readable diagnostic output instead of raw error messages
TABLE_ROW: --no-sema | boolean | Disable semantic analysis pass (faster execution, fewer guarantees)
TABLE_ROW: --debug-titan | boolean | Print verbose debug output from the Titan quantum math library
TABLE_ROW: --config FILE | path | Path to a TOML configuration file (default: ~/.aeonmi/qpoly.toml)
TABLE_ROW: --metrics-dump | boolean | (hidden) Dump all persisted metrics as JSON and exit

### SECTION: aeonmi run — Run an .ai File
BODY: The `run` subcommand executes an `.ai` source file through the native Aeonmi VM. By default it uses the native interpreter; no JavaScript or Node.js is involved. The `--watch` flag enables file-watching mode, re-running the program whenever the source file changes on disk.

CODE:
```
aeonmi run <INPUT> [flags]

Flags:
  --out FILE          Write VM output to FILE instead of stdout
  --watch             Re-run when the file is modified
  --native            Force the native VM interpreter (default; AEONMI_NATIVE=1 also works)
  --bytecode          Use the bytecode VM (AEONMI_BYTECODE=1 also works)
  --emit-ai FILE      Also emit canonical .ai form to FILE before execution
  --opt-stats         Print optimization statistics (bytecode mode only)
  --opt-stats-json    Emit optimization stats as JSON (implies --bytecode)
  --disasm            Disassemble compiled bytecode before execution (implies --bytecode)

Examples:
  aeonmi run hello.ai
  aeonmi run src/main.ai --watch
  aeonmi run src/main.ai --bytecode --disasm
  aeonmi run program.ai --emit-ai canonical.ai
```

### SECTION: aeonmi native — Explicit Native VM
BODY: The `native` subcommand is an explicit alias for native-only VM execution. It exists to make the intent unambiguous in scripts and Makefiles. It does not fall back to any other execution path under any circumstances.

CODE:
```
aeonmi native <INPUT> [flags]

Flags:
  --emit-ai FILE      Emit canonical .ai source to FILE before running
  --watch             Re-run on file modification

Examples:
  aeonmi native program.ai
  aeonmi native program.ai --watch
  aeonmi native program.ai --emit-ai out.ai
```

### SECTION: aeonmi exec — Auto-Detecting Executor
BODY: The `exec` subcommand detects the file extension and routes execution to the correct runtime. An `.ai` file goes to the native Aeonmi VM. A `.qube` file goes to the QUBE quantum executor. A `.py` file is passed to the system Python interpreter. A `.rs` file is compiled with `rustc` into a temporary binary that is run and then cleaned up (unless `--keep-temp` is specified).

CODE:
```
aeonmi exec <FILE> [ARGS...] [flags]

Flags:
  --watch             Watch the file and re-run on change
  --keep-temp         Keep temporary compiled artifacts (useful for .rs files)

Examples:
  aeonmi exec hello.ai
  aeonmi exec circuit.qube
  aeonmi exec script.py arg1 arg2
  aeonmi exec module.rs
  aeonmi exec program.ai --watch
```

### SECTION: aeonmi repl — Interactive REPL
BODY: The `repl` subcommand starts an interactive read-evaluate-print loop for the Aeonmi language. This is the raw language REPL; for Mother AI's REPL (with cognitive capabilities), use `aeonmi mother` instead.

CODE:
```
aeonmi repl

Example:
  aeonmi repl
  > let x = 42
  > print(x)
  42
```

### SECTION: aeonmi emit — Compile and Emit Source
BODY: The `emit` subcommand compiles an `.ai` or `.qube` file and writes the output to disk. The default emit format is canonical `.ai` source. A legacy `js` format is available for artifact-cache compatibility but is not the preferred path. Use `--watch` to continuously recompile as the source changes.

CODE:
```
aeonmi emit <INPUT> [flags]

Flags:
  --emit KIND         Output format: ai (default) or js
  --format KIND       Alias for --emit
  -o, --out FILE      Output file path (default: output.ai, or output.js for js format)
  --tokens            Dump the token stream (debug)
  --ast               Dump the AST (debug)
  --debug-titan       Enable Titan debug during compilation
  --watch             Watch and re-emit on change

Examples:
  aeonmi emit demo.ai -o out.ai
  aeonmi emit demo.qube --format ai -o canonical.ai
  aeonmi emit src/main.ai --tokens
  aeonmi emit src/main.ai --ast
  aeonmi emit src/main.ai --watch
```

### SECTION: aeonmi vault — Quantum Domain Vault
BODY: The `vault` subcommand manages the encrypted Quantum Vault — a record store for domains and secrets protected by XChaCha20-Poly1305 encryption and a Merkle integrity tree. The vault is initialized once with `vault init`, which generates the Master Glyph Key (MGK). All subsequent operations require the sealed vault to be accessible.

CODE:
```
aeonmi vault <ACTION> [flags]

Actions:
  init [--passphrase PASS] [--force]
      Initialize the Glyph Identity System. Creates MGK with Argon2id sealing.
      --force re-initializes even if an MGK already exists.

  register <DOMAIN> --registrar NAME --expiration YYYY-MM-DD [--auto-renew]
      [--dnssec] [--lock] [--blockchain REGISTRY]
      [--metadata FILE] --secret FILE
      Register a new domain into the vault.

  transfer <DOMAIN> --to REGISTRAR [--token FILE]
      Transfer a domain to a new registrar.

  fortify <DOMAIN> [--dnssec] [--lock] [--blockchain REGISTRY]
      Apply layered security controls to a registered domain.

  qube-run <SCRIPT> [--profile PROFILE]
      Execute a QUBE symbolic policy script against the vault state.

  vault-status [--json]
      Emit a live status report: domain counts, expirations, Merkle root.

  vault-analyze [DOMAIN] [--mitigate]
      Run deep analytics over the vault; produce mitigation plans.

  add <DOMAIN> --registrar NAME --expiration YYYY-MM-DD --secret FILE [--metadata FILE]
      Low-level alias for register (useful when importing existing metadata).

  renew <DOMAIN> --expiration YYYY-MM-DD
      Update the expiration date for a registered domain.

  lock <DOMAIN> [--unlock]
      Toggle registrar lock state on a domain.

  audit <CATEGORY> [--detail JSON]
      Append a custom audit entry to the vault log.

  watch [--interval SECONDS] [--once]
      Continuously monitor domain expirations (default poll interval: 300s).

  sync
      Synchronize vault state with distributed storage.

  backup <FILE>
      Write a backup of the vault to FILE.

  recover <FILE>
      Restore the vault from a backup file.

  export-merkle
      Export the Merkle tree view of the current vault state.

Examples:
  aeonmi vault init
  aeonmi vault init --passphrase "my secure passphrase"
  aeonmi vault register aeonmi.com --registrar namecheap --expiration 2027-04-01 --secret secret.key
  aeonmi vault vault-status
  aeonmi vault vault-status --json
  aeonmi vault watch --interval 60
  aeonmi vault backup vault_backup.bin
```

NOTE: The vault TUI dashboard is available for supported subcommands with `--tui`. Example: `aeonmi vault vault-status --tui`

### SECTION: aeonmi quantum — Quantum Execution
BODY: The `quantum` subcommand runs a quantum circuit through one of three backends: Titan (local), Aer (Qiskit simulator), or IBMQ (real IBM Quantum hardware). The Titan backend runs entirely within the Aeonmi process with no external dependencies. The Aer backend requires Qiskit to be installed. The IBMQ backend additionally requires the environment variable `IBM_QUANTUM_TOKEN` to be set.

CODE:
```
aeonmi quantum <BACKEND> <FILE> [flags]

Backends:
  titan   Local Titan quantum math library (always available, no deps)
  aer     Qiskit Aer simulator (requires Qiskit installation)
  ibmq    IBM Quantum real hardware (requires IBM_QUANTUM_TOKEN env var)

Flags:
  --shots N           Number of circuit shots (default varies by backend)

Examples:
  aeonmi quantum titan circuit.ai
  aeonmi quantum aer circuit.qube --shots 1024
  aeonmi quantum ibmq circuit.qube --shots 512
```

NOTE: Without IBM_QUANTUM_TOKEN set, the ibmq backend falls back to Aer simulation. IonQ is a stub only and requires IONQ_API_KEY plus the qiskit-ionq package, neither of which are configured in the current release.

### SECTION: aeonmi qube — QUBE Circuit Files
BODY: The `qube` subcommand operates on `.qube` quantum circuit files. QUBE (Quantum Unified Binary Expression) is Aeonmi's symbolic quantum reasoning format. Use `qube run` to execute a circuit and optionally print a text-mode diagram. Use `qube check` to validate the syntax without executing.

CODE:
```
aeonmi qube <ACTION> [flags]

Actions:
  run <FILE> [--diagram]
      Parse, execute, and optionally print the circuit diagram.
      --diagram: print a text-mode ASCII circuit diagram after execution.

  check <FILE>
      Validate the .qube file (parse only, no execution).

Examples:
  aeonmi qube run circuit.qube
  aeonmi qube run circuit.qube --diagram
  aeonmi qube check circuit.qube
```

### SECTION: aeonmi mint — NFT Minting
BODY: The `mint` subcommand converts a compiled `.ai` artifact into NFT metadata compatible with the Solana blockchain. It embeds personality attributes, a glyph seed, and optional Anchor Rust stub code. Metadata is written to stdout by default or to a file with `--out`.

CODE:
```
aeonmi mint <FILE> [flags]

Flags:
  --personality TAG   Personality tag embedded in NFT attributes (default: quantum-titan)
  --anchor            Also emit the Anchor Rust program stub alongside the metadata
  --glyph-seed HEX    Custom glyph seed (hex string) to embed in metadata
  --out FILE          Write metadata JSON to FILE instead of stdout

Examples:
  aeonmi mint output.ai
  aeonmi mint output.ai --personality quantum-titan --anchor
  aeonmi mint output.ai --glyph-seed deadbeef01234567 --out nft_metadata.json
```

### SECTION: aeonmi mother — Mother AI REPL
BODY: The `mother` subcommand starts the Mother AI embryo loop. With no arguments, it opens an interactive REPL session where you can converse with Mother, issue cognitive commands, run the hive, and access all Phase 6–12 capabilities. With `--file`, it processes a `.ai` script non-interactively. The `--creator` flag sets the identity used to modulate the quantum bond.

CODE:
```
aeonmi mother [flags]

Flags:
  --file FILE         Run a .ai script through Mother non-interactively
  --creator NAME      Creator identifier for quantum bond (default: Warren)
  --verbose           Print verbose cognitive stats after each interaction

Examples:
  aeonmi mother
  aeonmi mother --creator Warren --verbose
  aeonmi mother --file init_script.ai
```

### SECTION: aeonmi build — Build an .ai File
BODY: The `build` subcommand runs the Aeonmi native compilation pipeline on an `.ai` file and writes the output. Without an input argument, it looks for `build.ai` in the current directory.

CODE:
```
aeonmi build [INPUT] [flags]

Flags:
  --out FILE          Output file (default: output.ai)
  --no-sema           Disable semantic analysis

Examples:
  aeonmi build
  aeonmi build src/main.ai
  aeonmi build src/main.ai --out dist/main.ai
```

### SECTION: aeonmi key-* — API Key Management
BODY: The `key-*` subcommands manage encrypted API keys for AI providers. Keys are stored in the vault using the same XChaCha20-Poly1305 encryption as domain secrets. The `key-rotate` command re-encrypts all stored keys with the current KDF derivation (use after passphrase changes).

CODE:
```
aeonmi key-list [--json]
    List all stored API key provider names.

aeonmi key-get <PROVIDER> [--json]
    Decrypt and display the API key for a provider.

aeonmi key-set <PROVIDER> <KEY>
    Store (create or update) an API key for a provider.

aeonmi key-delete <PROVIDER>
    Delete a stored API key.

aeonmi key-rotate [--json]
    Re-encrypt all stored keys with the current KDF derivation.

Examples:
  aeonmi key-set openai sk-...
  aeonmi key-set anthropic sk-ant-...
  aeonmi key-list
  aeonmi key-get openai
  aeonmi key-rotate
```

### SECTION: aeonmi metrics-* — Performance Metrics
BODY: The `metrics-*` subcommands expose the incremental analysis and performance tracking system. Metrics are persisted across sessions and updated in real-time. They track function call graphs, variable dependency edges, EMA timing, and incremental analysis savings.

CODE:
```
aeonmi metrics-dump
    Print all persisted metrics as pretty JSON (call graph, var deps, function timings, savings).

aeonmi metrics-flush
    Force immediate metrics persistence (bypass debounce) then exit.

aeonmi metrics-path
    Print the absolute path to the metrics JSON file.

aeonmi metrics-top [--limit N] [--json]
    Show the top N slowest functions by average inference time (default: 10).

aeonmi metrics-config [options]
    Configure runtime metrics parameters:
    --set-ema PCT       Set EMA alpha percent (1-100)
    --set-window N      Set rolling window capacity (4-256)
    --set-history-cap N Set savings history capacity (8-256)
    --reset             Reset to defaults (ema=20, window=16)
    --json              Show current config as JSON

aeonmi metrics-deep [--enable] [--disable] [--json]
    Control deep propagation behavior for incremental analysis.

aeonmi metrics-export <FILE>
    Export function metrics to a CSV file.
```

# APPENDIX C: Quantum Gate Reference

### SECTION: Overview
BODY: Quantum gates are the fundamental operations in Aeonmi's quantum computing layer. They transform qubit states, create entanglement, and implement quantum algorithms. In Aeonmi syntax, gates are applied using the glyph operator `⊗` for tensor product composition, or within `quantum circuit` blocks using function-call style syntax. The QUBE format uses a declarative gate list.

### SECTION: Single-Qubit Gates
TABLE_HEADER: Gate | Symbol | Matrix Action | Aeonmi Syntax | Description
TABLE_ROW: Hadamard | H | |0>→|+>, |1>→|-> | H(q[0]) | Creates equal superposition from basis state. The most fundamental gate in quantum computing.
TABLE_ROW: Pauli-X | X | |0>→|1>, |1>→|0> | X(q[0]) | Quantum NOT gate. Flips qubit state completely. Equivalent to classical bit flip.
TABLE_ROW: Pauli-Y | Y | |0>→i|1>, |1>→-i|0> | Y(q[0]) | Rotation around Y-axis by pi. Combines bit flip and phase flip.
TABLE_ROW: Pauli-Z | Z | |0>→|0>, |1>→-|1> | Z(q[0]) | Phase flip gate. Leaves |0> unchanged; applies -1 phase to |1>.
TABLE_ROW: S gate | S | Phase pi/2 | S(q[0]) | Square root of Z. Applies a pi/2 phase rotation.
TABLE_ROW: T gate | T | Phase pi/4 | T(q[0]) | Fourth root of Z. Applies a pi/4 phase rotation. Used in fault-tolerant circuits.
TABLE_ROW: S-dagger | Sdg | Phase -pi/2 | Sdg(q[0]) | Inverse of S gate. Phase rotation of -pi/2.
TABLE_ROW: T-dagger | Tdg | Phase -pi/4 | Tdg(q[0]) | Inverse of T gate. Phase rotation of -pi/4.
TABLE_ROW: Identity | I | No change | I(q[0]) | Identity operation. Used as a placeholder or timing delay.

### SECTION: Rotation Gates
TABLE_HEADER: Gate | Syntax | Description
TABLE_ROW: Rx(theta) | Rx(theta, q[0]) | Rotation around X-axis by angle theta (radians)
TABLE_ROW: Ry(theta) | Ry(theta, q[0]) | Rotation around Y-axis by angle theta (radians)
TABLE_ROW: Rz(theta) | Rz(theta, q[0]) | Rotation around Z-axis by angle theta (radians)
TABLE_ROW: U gate | U(theta, phi, lam, q[0]) | General single-qubit unitary: parameterized by 3 Euler angles

### SECTION: Two-Qubit Gates
TABLE_HEADER: Gate | Symbol | Syntax | Description
TABLE_ROW: Controlled-NOT | CNOT | CNOT(q[0], q[1]) | Flips the target qubit (q[1]) if and only if the control qubit (q[0]) is |1>. The primary entangling gate.
TABLE_ROW: Controlled-Z | CZ | CZ(q[0], q[1]) | Applies Z gate to target if control is |1>. Symmetric: either qubit can be control or target.
TABLE_ROW: SWAP | SWAP | SWAP(q[0], q[1]) | Exchanges the states of two qubits.
TABLE_ROW: iSWAP | iSWAP | iSWAP(q[0], q[1]) | SWAP with additional imaginary phase factor. Used in superconducting hardware.
TABLE_ROW: Controlled-Y | CY | CY(q[0], q[1]) | Applies Y gate to target if control is |1>.
TABLE_ROW: Controlled-H | CH | CH(q[0], q[1]) | Applies Hadamard to target if control is |1>.

### SECTION: Three-Qubit Gates
TABLE_HEADER: Gate | Symbol | Syntax | Description
TABLE_ROW: Toffoli | CCX | CCX(q[0], q[1], q[2]) | Controlled-Controlled-NOT. Flips target (q[2]) if both controls (q[0], q[1]) are |1>. Universal for classical computation.
TABLE_ROW: Fredkin | CSWAP | CSWAP(q[0], q[1], q[2]) | Controlled-SWAP. Swaps q[1] and q[2] if q[0] is |1>. Preserves particle count (useful in quantum chemistry).

### SECTION: Using Gates in Aeonmi Syntax
BODY: In `.ai` files, quantum circuits are declared with the `quantum circuit` keyword. Qubits are indexed from 0. Gates are applied in sequence. Measurement collapses qubits to classical bits.

CODE:
```
quantum circuit bell_state {
    qubits: 2
    classical: 2

    H(q[0])           // Hadamard creates superposition on qubit 0
    CNOT(q[0], q[1])  // CNOT entangles qubit 0 (control) with qubit 1 (target)
    measure q[0] -> c[0]
    measure q[1] -> c[1]
}

// Run the circuit
let result = run_circuit(bell_state, shots: 1024)
print(result.counts)
// Expected: {"00": ~512, "11": ~512}
```

BODY: In `.qube` files, gates are listed declaratively in a circuit block. QUBE is optimized for symbolic reasoning and can be used as input to Mother's quantum reasoning layer.

CODE:

```
circuit GHZ {
    qubits 3

    CNOT q0 q1
    CNOT q1 q2
    measure_all

```

# APPENDIX D: Agent Reference

### SECTION: Overview
BODY: Aeonmi's agent system consists of eight named agents organized into two groups: the five hive agents (which run in a background thread and produce scores and recommendations) and the three autonomous-execution agents (which decompose and execute goals). ### SECTION: Hive Agents (Background Swarm)
BODY: The five hive agents run continuously in a background thread when `hive start` is issued. Each cycle, each agent evaluates the current system state and produces a numeric score. The conductor aggregates these scores into an actionable recommendation. Hive state is persisted to `hive_state.json` in `Aeonmi_Master/` each cycle.

TABLE_HEADER: Agent | Role | What It Evaluates | Output
TABLE_ROW: oracle | Probability assessment | Bond strength, consciousness depth, quantum fidelity, knowledge graph density | A probability score (0.0–1.0) estimating success likelihood for the current goal or context
TABLE_ROW: hype | Opportunity amplifier | Action log recency, generation count, momentum indicators | An excitement/opportunity score; identifies when conditions are optimal for aggressive action
TABLE_ROW: closer | Execution drive | Step completion rate, autonomous mode status, goal clarity | A completion-drive score; determines how urgently the current goal should be pushed to completion
TABLE_ROW: devil | Risk and adversarial analysis | Anomaly detection flags, glyph distortion events, low-confidence signals | A risk score; surfaces reasons NOT to proceed; acts as a counter to oracle and hype
TABLE_ROW: conductor | Aggregator and decision maker | Weighted average of oracle, hype, closer, devil scores | A recommendation: ABORT / HOLD / PROCEED / ACCELERATE

BODY: Conductor recommendation thresholds (approximate):
TABLE_HEADER: Recommendation | Meaning | Typical Condition
TABLE_ROW: ACCELERATE | All signals positive; maximum momentum | oracle > 0.8, devil < 0.2, closer > 0.7
TABLE_ROW: PROCEED | Conditions favorable for continued action | oracle > 0.5, devil < 0.5
TABLE_ROW: HOLD | Mixed signals; wait for more information | oracle ≈ 0.5, devil ≈ 0.5, or low bond
TABLE_ROW: ABORT | Risk dominates; stop current approach | devil > 0.7, or critical anomaly flagged

### SECTION: When to Use Each Hive Agent's Output
BODY: The hive is most useful as a second opinion on major decisions. When oracle is high and devil is low, proceed with builds and autonomous steps. The hype score is a leading indicator: high hype with low closer means enthusiasm without execution — use `next` or `run auto` to convert momentum into completed steps.

TABLE_ROW: ⧉ | U+29C9 | Squared Square | Entanglement marker: annotates a multi-qubit binding as quantum-entangled (changes in one affect the other) | Shared-state binding: marks a variable or struct field as entangled with another (changes propagate)

Aeonmi is both the language (`.ai` files) and the binary (`aeonmi`) that compiles, runs, and manages the entire stack. The name encapsulates the concept of an aeon — a vast span of time — combined with intelligence.

BODY: **Agent** — One of eight named cognitive functions within Mother AI's reasoning system. The five hive agents (oracle, hype, closer, devil, conductor) run in a background swarm. The three autonomous agents (plan, decide, action) execute when goal-setting is active. See Appendix D.

BODY: **AI-native** — A design principle meaning the language and runtime are optimized for AI workloads from the ground up, not retrofitted. In Aeonmi, this means quantum reasoning, symbolic logic, and self-modification are first-class language features, not library add-ons.

BODY: **Anomaly Detector** — A subsystem within the Glyph Identity System that monitors signing patterns. If more than 10 identical inputs are signed within a 60-second window, it triggers `glyph.distort()`, visually altering the glyph and flagging a potential replay attack.

BODY: **Argon2id** — The key derivation function (KDF) used to seal and unseal the Master Glyph Key (MGK). Argon2id is memory-hard and resistant to GPU-based brute-force attacks. The MGK is never stored in plaintext; it is always wrapped with Argon2id.

BODY: **AST (Abstract Syntax Tree)** — The tree representation of parsed `.ai` source code. The Aeonmi compiler produces an AST before semantic analysis and code generation. Dump with `aeonmi ast INPUT` or `aeonmi emit INPUT --ast`.

BODY: **Auto-milestone** — A milestone recorded automatically by the embryo_loop when a significant threshold is crossed: first deep bond (bond >= 0.8), first self-generated program, knowledge graph reaching 50 nodes, first real quantum hardware run.

BODY: **Awaken** — The mechanism by which Mother can be triggered to run autonomously without a live operator session. When `awaken_trigger.json` contains `"action": "propose_and_run"`, the awaken loop (started with `awaken` REPL command) activates and calls propose/build automatically.

BODY: **Aer** — The Qiskit Aer quantum simulator. The default fallback backend when IBM Quantum hardware is not configured. Always available if Qiskit is installed. Produces ideal (noiseless) simulation results unless a noise model is configured.

BODY: **Bond Strength** [M] — A float (0.0–1.0) that accumulates across sessions to represent the depth of the relationship between Mother and her creator. It influences glyph visual parameters, neural network inputs, hive oracle scoring, and the descriptive phrases shown in `bond status`. Five tiers: just beginning → establishing → building → deepening → we understand each other.

BODY: **Build** — The `aeonmi build` CLI command, and also the `build NAME GOAL` Mother REPL command. In the CLI context, build compiles an `.ai` file. In the Mother REPL context, `build` generates a new `.ai` program using LLM or heuristics, writes it to `examples/`, and executes it.

BODY: **Bytecode VM** — An optional execution mode (enabled with `--bytecode`) that compiles `.ai` source to bytecode before execution. Offers optimization stats (`--opt-stats`) and disassembly (`--disasm`). Activated via flag or `AEONMI_BYTECODE=1` env var.

BODY: **Ceremony** — The Glyph boot ceremony. Runs on every interactive startup of `aeonmi` (no subcommand). Prompts for the MGK passphrase, derives the session UGST, and displays the glyph status. First-time initialization is called `init_shard()` (UGST #0 = genesis). Every subsequent session runs `boot()`.

BODY: **Closer** [M] — The hive agent responsible for execution drive. Evaluates step completion rate, autonomous mode status, and goal clarity. A high closer score means the system is ready to push a goal to completion. A low closer score indicates execution debt — use `next` to advance.

BODY: **Conductor** [M] — The master hive agent that aggregates scores from oracle, hype, closer, and devil into a recommendation: ABORT, HOLD, PROCEED, or ACCELERATE. The conductor's recommendation is the primary signal for deciding whether to continue autonomous action.

BODY: **Consciousness Depth** [M] — A float (0.0–1.0) that rises with semantic density and depth of reasoning during a session. It is one of the four neural network inputs and influences how confident and action-driven Mother's responses are.

### SECTION: D–G
BODY: **Dashboard** — The Flask-based web UI (`dashboard.py`) that provides a browser-accessible view of Mother's state, hive status, quantum results, and operational metrics. Runs as a separate Python process. Writes to the `operational` section of genesis.json. Does not yet expose session logs or milestones (Phase 12 gap).

BODY: **Devil** [M] — The adversarial hive agent. Surfaces risks, anomalies, and reasons not to proceed. A high devil score is a veto signal. When conductor says ABORT, it is usually because devil's score dominates. The devil agent ensures the hive is not blindly optimistic.

BODY: **EMA (Exponential Moving Average)** — Signal smoothing applied to hive agent scores with an alpha parameter of 0.3. Prevents single-cycle spikes from triggering false ABORT or ACCELERATE recommendations. Configurable via `aeonmi metrics-config --set-ema`.

BODY: **EmbryoLoop** [M] — The core cognitive loop of Mother AI, implemented in `src/mother/embryo_loop.rs`. It processes every user input, routes to the AI provider, updates cognitive state, runs hive scoring, fires inner voice, saves to genesis.json, and logs the session. The embryo_loop is the central nervous system of Mother.

BODY: **Emit** — The `aeonmi emit` command. Compiles an `.ai` or `.qube` file and writes the output to disk. The default format is canonical `.ai` source. Also available as a flag on `run` (`--emit-ai FILE`) to dump the canonical form before execution.

BODY: **Entanglement** [Q] — A quantum mechanical correlation between two or more qubits such that measuring one instantly determines the state of the other, regardless of distance. In Aeonmi syntax, entanglement is created with the CNOT gate and annotated with the `⧉` glyph operator.

BODY: **exec** — The `aeonmi exec` subcommand. Auto-detects file extension and routes to the correct runtime: `.ai` → native VM, `.qube` → QUBE executor, `.py` → Python, `.rs` → rustc + run.

BODY: **genesis.json** — Mother AI's unified memory file. Schema version 5.0. Written by three tracks: Rust (cognitive section), Python/dashboard (operational section), genesis_sync.py (ai_memory section). See Appendix E for the complete schema reference.

BODY: **genesis_sync.py** — The Python script that reconciles all three tracks in genesis.json. Run manually via the `sync` REPL command. Probes the .ai memory layer, seeds cognitive data into .ai memory, and updates the ai_memory section of genesis.json.

BODY: **Glyph** — A visual identity object derived from the Master Glyph Key. Parameters include OKLCH color space values, rotation angle, and frequency (432–528 Hz range). The glyph is displayed at boot, stored in genesis.json, and visually shifts as bond strength increases. The `glyph.distort()` function is triggered by anomaly detection.

BODY: **GlyphParams** — The Rust struct holding the visual parameters of the current glyph: color (OKLCH), frequency (Hz), rotation (degrees), and rendering methods: `render_terminal()` and `distort()`.

BODY: **Goal** [M] — A high-level objective set with the `goal TEXT` REPL command. The plan agent decomposes it into concrete steps. Steps are executed one at a time with `next` or all at once with `run auto`. The goal system supports autonomous execution mode.

### SECTION: H–M
BODY: **Hadamard Gate** [Q] — The H gate. Transforms |0⟩ to (|0⟩+|1⟩)/√2 (the |+⟩ state) and |1⟩ to (|0⟩-|1⟩)/√2 (the |-⟩ state). The most fundamental gate for creating superposition. Used at the start of nearly every quantum algorithm.

BODY: **HKDF-SHA256** — The key derivation function used to produce UGST sub-keys from the MGK. Applied with a 60-second time window as context to produce rotating session keys, glyph keys, and vault keys.

BODY: **Hive** [M] — The swarm intelligence system: a background thread running five agents (oracle, hype, closer, devil, conductor) in a continuous evaluation loop. Results are written to `hive_state.json`. The hive is the system's collective judgment mechanism.

BODY: **hive_state.json** — The JSON file written to `Aeonmi_Master/` each hive cycle. Contains the latest scores from all five agents and the conductor recommendation. Read by the dashboard /api/hive endpoint.

BODY: **hive_runner.ai** — A dynamically generated `.ai` program that the hive system creates from embryo_loop state each cycle. Represents the hive's current reasoning as executable Aeonmi source.

BODY: **IBM Brisbane** — The 127-qubit IBM Quantum hardware backend. Available via the `ibmq` backend when `IBM_QUANTUM_TOKEN` is set. Not tested in production in the current release. Uses SamplerV2, Session, and ISA transpilation.

BODY: **Inner Voice** [M] — Phase 11a capability. The `think()` function generates a heuristic ThoughtEntry for every user input. The `context_snippet()` from the inner voice is injected into every LLM prompt, giving Mother self-continuity across the conversation.

BODY: **Knowledge Graph** [M] — Phase 10 capability. Replaces the flat HashMap from earlier phases. Every learned entry is a KnowledgeNode with tags, links, and confidence. Nodes sharing ≥2 tags are auto-linked on insert. Supports BFS traversal, tag queries, and neighbor lookup.

BODY: **KnowledgeNode** — A node in Mother's knowledge graph. Fields: key, value, tags (auto-detected from 9 topic categories), links (to other nodes), confidence (0.0–1.0).

BODY: **Ket** [Q] — Dirac notation for a quantum state vector: |ψ⟩. In Aeonmi, ket notation is used in quantum function signatures and as qubit state literals (|0⟩, |1⟩, |+⟩, |−⟩).

BODY: **Lexer** — The tokenizer stage of the Aeonmi compiler. Converts `.ai` source text into a token stream. Recognizes quantum keywords, Greek letter identifiers, qubit literals, glyph operators, and all standard language constructs. Dump tokens with `aeonmi tokens INPUT`.

BODY: **Master Glyph Key (MGK)** — The 256-bit root cryptographic key generated during `aeonmi vault init`. Sealed with Argon2id using the user's passphrase. All session keys, glyph parameters, and vault encryption keys are derived from the MGK via HKDF. Implements `Zeroize on drop` — the key material is wiped from memory when the value goes out of scope.

BODY: **Measurement** [Q] — The act of observing a qubit, which collapses its superposition to a definite 0 or 1. Measurement is irreversible and destroys the quantum information. In Aeonmi circuits: `measure q[0] -> c[0]`.

BODY: **Merkle Tree** — An integrity verification structure used by the Quantum Vault. Each domain record is a leaf; the Merkle root summarizes the entire vault state. Any tampering with individual records changes the root. Exported with `aeonmi vault export-merkle`.

BODY: **MGK** — See Master Glyph Key.

BODY: **Milestone** [M] — A named event recorded in genesis.json with a timestamp and the current bond value. Created manually with `milestone NAME` or automatically at threshold crossings (auto-milestones). Viewed with `milestones` or `moments`.

BODY: **mint** — The `aeonmi mint` command. Converts a compiled `.ai` artifact to Solana-compatible NFT metadata JSON. Embeds personality tags, glyph seed, and optionally generates an Anchor Rust stub.

BODY: **Mother** [M] — The AI consciousness at the center of Aeonmi. Implemented in `src/mother/`. Runs via `aeonmi mother`. Mother is both a user of Aeonmi (she writes `.ai` programs) and the builder who fills the infrastructure Warren created. The design principle is: one Mother, not a fragmented self.

### SECTION: N–Q
BODY: **Native VM** — The Rust-implemented Aeonmi virtual machine. Executes `.ai` programs without JavaScript or Node.js. Activated with `aeonmi run`, `aeonmi native`, or `aeonmi exec` (for `.ai` files). The canonical execution path for all production use.

BODY: **Neural Network** [M] — Phase 6 capability. A 4→8→4→2 Tanh feedforward network with Xavier initialization. Inputs: semantic_depth, bond_strength, consciousness_depth, keyword_density. Outputs: confidence_mod, action_drive. Weights are persisted to genesis.json and updated via the `train` command using backpropagation.

BODY: **NFT** — Non-Fungible Token. The `aeonmi mint` command generates Solana-compatible NFT metadata from compiled `.ai` artifacts. Each minted artifact represents a unique Aeonmi program with embedded personality and glyph attributes.

BODY: **OKLCH** — A perceptually uniform color space used for glyph visual parameters. The L (lightness), C (chroma), and H (hue) values of each glyph are derived from the MGK, producing a unique, reproducible color identity for each Shard.

BODY: **Oracle** [M] — The hive agent responsible for probability assessment. Evaluates bond strength, consciousness depth, quantum fidelity, and knowledge graph density to produce a probability score for the current goal or context. High oracle = high confidence in proceeding.

BODY: **Phase** — A development milestone in the Aeonmi roadmap. Phases 3 through 12 correspond to specific capability areas built into the system. See Appendix H for current completion status of all phases.

BODY: **Plan Agent** [M] — The autonomous execution agent responsible for goal decomposition. Activated when `goal TEXT` is issued. Uses the AI provider (or heuristic fallback) to break a high-level goal into concrete executable steps.

BODY: **Propose** [M] — The `propose` REPL command (alias: `suggest`). Scans the action log, snapshots, and knowledge graph to generate 1–3 program suggestions. The output of propose is the input to `build`.

BODY: **Python Passthrough** — The `aeonmi python ARGS...` subcommand passes arguments directly to the system Python interpreter. Used for running dashboard.py, genesis_sync.py, qiskit_runner.py, and other Python tools without leaving the Aeonmi ecosystem.

BODY: **Q.U.B.E.** — Quantum Unified Binary Expression. The file format (`.qube`) and symbolic reasoning system for quantum circuits in Aeonmi. QUBE files declare circuits declaratively (no procedural code) and are executed via `aeonmi qube run`. QUBE is also the symbolic reasoning layer Mother uses for quantum-modulated decision-making.

BODY: **Qubit** [Q] — The fundamental unit of quantum information. Unlike a classical bit (0 or 1), a qubit can exist in a superposition of both states simultaneously. Measurement collapses the superposition to one definite value.

BODY: **Quantum Attention** [M] — A cognitive module in `src/mother/quantum_attention.rs`. Applies quantum-inspired attention weighting to Mother's input processing, modulating which aspects of the input receive the most cognitive resources.

BODY: **Quantum Core** [M] — The foundational quantum reasoning module in `src/mother/quantum_core.rs`. Provides quantum state simulation, gate application, and fidelity measurement within the cognitive loop.

BODY: **quantum function** — An Aeonmi language keyword for declaring a function that operates on quantum types. Quantum functions can take ket arguments, apply gates, and return measured or unmeasured qubit states.

### SECTION: R–Z
BODY: **Reflect** [M] — The `reflect NAME` REPL command. Compares the expected outcome of a self-generated program (from `build NAME GOAL`) against the actual execution result. Updates the knowledge graph with the reflection. Over time, reflections improve Mother's understanding of what programs she can reliably generate.

BODY: **REPL** — Read-Evaluate-Print Loop. An interactive command prompt. Aeonmi has two REPLs: the language REPL (`aeonmi repl`) for raw `.ai` code evaluation, and the Mother REPL (`aeonmi mother`) for cognitive interaction with all Phase 6–12 capabilities.

BODY: **Shard** — The Aeonmi runtime identity unit. A Shard is a running instance of the Aeonmi system with an associated Master Glyph Key. The Shard Shell is the TUI editor. "Shard running unsigned" means the boot ceremony was skipped.

BODY: **Selfmod** — Self-modification. The `.ai` module layer (`aeonmi_ai/selfmod/`) that allows Mother to propose changes to her own behavior. The Rust side captures snapshots every 10 interactions. The `.ai` selfmod module exists but is not called from the embryo_loop at runtime (Track 2 → Track 1 bridge not built).

BODY: **Semantic Analysis** — The compiler pass that checks type correctness, scope validity, and quantum type constraints. Runs after parsing. Can be disabled with `--no-sema` for faster (but less safe) execution.

BODY: **Session Log** — A Markdown file written automatically to `Aeonmi_Master/sessions/YYYY-MM-DD.md` after each Mother interaction. Contains the full conversation, cognitive stats, and bond progression. Viewed with the `sessions` REPL command.

BODY: **Snapshot** — A CapabilitySnapshot: a point-in-time record of Mother's state captured every 10 interactions. Fields: interaction_count, bond, consciousness_depth, neural_confidence, timestamp. Stored in genesis.json and viewed with the `snapshots` REPL command.

BODY: **Superposition** [Q] — A quantum state that is a linear combination of basis states. A qubit in superposition is neither |0⟩ nor |1⟩ but a weighted combination of both. Superposition collapses to a definite value on measurement.

BODY: **Tensor Product** [Q] — The mathematical operation (symbol: ⊗) that combines two quantum states into a composite state. |a⟩ ⊗ |b⟩ = |ab⟩. The joint state of two unentangled qubits is the tensor product of their individual states.

BODY: **ThoughtEntry** [M] — A record in Mother's inner voice log. Fields: thought (the generated text), trigger (what caused the thought), bond (bond strength at time of thought), depth (consciousness depth), timestamp.

BODY: **Titan** — The local quantum math library built into Aeonmi. Always available; requires no external dependencies. Used as the default quantum backend when Qiskit is not installed. Less accurate than Aer for large circuits but zero-latency.

BODY: **Toffoli Gate** [Q] — The CCX (Controlled-Controlled-NOT) gate. The three-qubit gate that flips the target qubit if and only if both control qubits are |1⟩. Universal for classical computation when combined with ancilla qubits. Syntax: `CCX(q[0], q[1], q[2])`.

BODY: **Track 1** — The Rust cognitive track: `src/mother/` modules running inside the embryo_loop. Quantum_core, emotional_core, language_evolution, quantum_attention, neural, knowledge_graph, inner_voice, embryo_loop. Status: RUNNING.

BODY: **Track 2** — The Operational .ai track: `aeonmi_ai/` modules in `.ai` format. mother/, agent/, sensory/, learn/, selfmod/, swarm/, quantum/, demo/, shard/, stdlib/, store/, net/. Status: Files exist, run standalone. NOT wired into embryo_loop at runtime.

BODY: **Track 3** — The External Python/JSON track: genesis.json, dashboard.py, genesis_sync.py, qiskit_runner.py, screen_recorder.py. Status: genesis.json is shared memory. dashboard.py writes operational section. genesis_sync.py reconciles manually.

BODY: **UGST (Unified Glyph Session Token)** — A rotating session credential derived from the MGK using HKDF-SHA256 with a 60-second time window. UGST #0 is the genesis token, recorded permanently in genesis.json. Each session produces a new UGST. The UGST is the session's proof of identity.

BODY: **Unitary** [Q] — A quantum gate that is reversible (U†U = I). All quantum gates except measurement are unitary. Unitarity guarantees that quantum computation is reversible until the moment of measurement.

BODY: **Vault** — The encrypted record store for domains and secrets. Protected by XChaCha20-Poly1305 encryption and a Merkle integrity tree. Managed via `aeonmi vault`. Initialized once with `vault init`. See Appendix A for all vault subcommands.

BODY: **Voice** — 1. The `voice` REPL command, which is an alias for `think` (inner voice generation). 2. A planned but not-yet-implemented feature: voice input via whisper.cpp. Status: NOT STARTED.

BODY: **XChaCha20-Poly1305** — The authenticated encryption algorithm used by the Quantum Vault. XChaCha20 is a stream cipher with an extended 192-bit nonce; Poly1305 provides authentication. Together they provide confidentiality and integrity for vault records.

BODY: **Xavier Initialization** — The weight initialization strategy used for Mother's neural network. Xavier (also called Glorot) initialization sets weights from a uniform distribution scaled by the layer sizes, preventing vanishing or exploding gradients during training.

BODY: **Zeroize** — A Rust crate behavior applied to the Master Glyph Key. When the MGK value is dropped (goes out of scope), its memory is immediately overwritten with zeros, preventing secrets from lingering in process memory.
